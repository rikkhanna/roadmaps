1 The Big Picture

Python packaging solves two problems: isolation (each project gets its own dependencies) and distribution (sharing your code as a reusable library or app).

System Python
→
venv / conda env
→
Isolated Python
→
pip install pkgs
→
Project runs ✓
venv / conda
Isolation

Creates a private Python + site-packages per project

pip / conda
Installation

Downloads packages from PyPI or conda-forge

pyproject.toml
Configuration

Declares your project's metadata and dependencies

💡 Analogy

Think of each virtual environment as a clean room. Your project runs in its own clean room with exactly the packages it needs — no conflicts with other projects. pip / conda are the delivery service that brings packages into the room. pyproject.toml is the manifest listing what's needed.

2 venv

venv is Python's built-in module (Python 3.3+) for creating lightweight virtual environments. It copies (or symlinks) the Python binary and creates an isolated site-packages folder.

Create & Activate

terminal (bash / zsh) 
# Create a new virtual environment in ./venv
python3 -m venv venv

# Activate it (changes your PATH so 'python' and 'pip' point to the venv)
source venv/bin/activate          # macOS / Linux
venv\Scripts\activate             # Windows

# Verify: the prompt changes and 'which python' shows the venv path
which python                       # → /path/to/project/venv/bin/python

# Deactivate (returns to system Python)
deactivate

What's Inside a venv?

venv/
├── bin/                  # Executables (python, pip, activate script)
│   ├── python3           # Symlink to system Python binary
│   ├── pip
│   └── activate          # The script you source to activate
├── lib/
│   └── python3.x/
│       └── site-packages/ # All installed packages land here
└── pyvenv.cfg            # Config: which Python, include-system-site-packages
📌 pyvenv.cfg explained

pyvenv.cfg stores the home path (where the base Python lives) and include-system-site-packages = false (by default your venv is isolated from system packages). Pass --system-site-packages to the create command if you want to share them.

Common venv Options

FlagEffectExample
--system-site-packagesInclude packages from system Pythonpython3 -m venv --system-site-packages venv
--without-pipSkip bundled pip (use if pip not needed)python3 -m venv --without-pip venv
--prompt NAMECustom shell prompt labelpython3 -m venv --prompt myapp venv
--upgrade-depsUpgrade pip + setuptools after creatingpython3 -m venv --upgrade-deps venv
--copiesCopy binaries instead of symlinkingpython3 -m venv --copies venv
⚠ Never commit venv/

Add venv/ (or whatever you named it) to .gitignore. Environments are machine-specific — teammates recreate them from requirements.txt or pyproject.toml.

Concept: Why isolation matters

❌ No isolation (global install)
  • Project A needs requests==2.28
  • Project B needs requests==2.31
  • Installing one breaks the other
  • Running pip install as root affects the OS
✅ With venv
  • Project A → venv_a/ → requests 2.28
  • Project B → venv_b/ → requests 2.31
  • Zero conflicts — each env is a separate directory
  • Delete env to clean up completely

3 pip

pip is Python's standard package installer. It downloads packages from PyPI (Python Package Index) and installs them into the currently active environment.

Core Commands

terminal 
# Install a package
pip install requests
pip install "requests==2.31.0"      # exact version
pip install "requests>=2.28,<3"     # version range

# Install from requirements file
pip install -r requirements.txt

# Install current project (editable)
pip install -e .                   # see Editable Installs section

# List installed packages
pip list
pip show requests                  # detailed info: version, location, deps

# Upgrade a package
pip install --upgrade requests

# Uninstall
pip uninstall requests

# Freeze: dump currently installed packages to a file
pip freeze > requirements.txt

# Check for outdated packages
pip list --outdated

requirements.txt — The Dependency File

A plain text file listing packages (one per line) for pip install -r to install. It's the simplest way to share "what your project needs."

requirements.txt 
# Pinned exact versions — reproducible but rigid
numpy==1.26.4
pandas==2.2.1
scikit-learn==1.4.2
matplotlib==3.8.4

# Version ranges — flexible but may break on updates
requests>=2.28,<3
flask~=3.0      # ~= means >=3.0 and ==3.*  (compatible release)

Version Specifiers Explained

SpecifierMeaningExample
==Exact matchrequests==2.31.0
>=Minimum versionrequests>=2.28
<=Maximum versionrequests<=3.0
!=Exclude a versionrequests!=2.30.0
~=Compatible releaserequests~=2.28 → >=2.28,==2.*
^Caret (Poetry/TOML only)^2.28 → >=2.28,<3

pip install from different sources

# From PyPI (default)
pip install numpy

# From a Git repo
pip install git+https://github.com/user/repo.git
pip install git+https://github.com/user/repo.git@v1.2.3   # specific tag

# From a local directory (the directory must have pyproject.toml or setup.py)
pip install ./my-local-package/

# From a .whl or .tar.gz file
pip install package-1.0.0-py3-none-any.whl

# Private index / mirror
pip install --index-url https://my-private-pypi.com/simple/ mypackage

Editable Installs — pip install -e .

An editable install (development install) installs the package without copying files. Instead, Python is told "treat this directory as an installed package." Changes to the source files are immediately reflected — no reinstall needed.

Regular install
pip install .
# Copies source into site-packages
# Changes require reinstall
Editable install
pip install -e .
# Creates a .pth file pointing here
# Edit code → changes live immediately
💡 When to use -e

Always use pip install -e . when actively developing a package. You get import mypackage from anywhere in the venv while editing the source. Switch to a regular install for testing your built distribution.

4 conda

conda is a cross-language package and environment manager from Anaconda. It can install Python itself (not just Python packages) and handles binary dependencies like BLAS, CUDA, and C libraries — making it indispensable for data science / ML.

Core Commands

terminal 
## ── Environments ──────────────────────────────────────────────

# Create env with specific Python version
conda create -n myenv python=3.11

# Create from environment.yml (the conda equivalent of requirements.txt)
conda env create -f environment.yml

# Activate / deactivate
conda activate myenv
conda deactivate

# List all environments
conda env list              # also: conda info --envs

# Remove an environment
conda env remove -n myenv

## ── Packages ──────────────────────────────────────────────────

# Install packages (from defaults + conda-forge)
conda install numpy pandas scikit-learn
conda install -c conda-forge lightgbm  # use conda-forge channel

# Update a package
conda update numpy
conda update --all

# Remove a package
conda remove numpy

# List packages in current env
conda list

# Search for a package
conda search scikit-learn

## ── Export / Recreate ─────────────────────────────────────────

# Export environment (cross-platform, omit platform-specific builds)
conda env export --from-history > environment.yml

# Full export (locks exact builds — platform specific)
conda env export > environment_lock.yml

environment.yml

environment.yml 
name: myenv

channels:
  - conda-forge       # community-maintained, often more up-to-date
  - defaults           # Anaconda's own channel

dependencies:
  - python=3.11        # conda installs Python itself!
  - numpy>=1.26
  - pandas
  - scikit-learn
  - matplotlib
  - pip:               # packages only on PyPI, install via pip
    - some-pypi-only-package==1.2.3
💡 conda-forge channel

Always add conda-forge as your first channel. It's community-maintained, has more packages, and more frequent updates than the defaults channel. Run conda config --add channels conda-forge to make it global.

Mini-conda vs Anaconda

Anaconda (~3 GB)
  • Full distribution with 250+ pre-installed packages
  • Includes Jupyter, Spyder, Navigator GUI
  • Good for beginners who want "everything"
Miniconda (~60 MB)
  • Minimal: just conda + Python
  • Install only what you need
  • Recommended for developers

5 pip vs conda

Both are package managers but with different scopes and strengths. They are often used together.

Featurepipconda
Package source PyPI (~550k packages) conda-forge / defaults (~25k packages)
Language scope Python only Python, R, C, C++, Fortran, CUDA, …
Environment mgmt Uses venv (separate tool) Built-in (conda create/activate)
Binary deps (BLAS, CUDA) Relies on OS or wheels Handles natively — great for GPU/ML
Dependency solver Fast (pip resolver) Thorough SAT solver — slower but consistent
Installs Python itself? No — uses existing Python Yes — can create envs with any Python version
Config file requirements.txt / pyproject.toml environment.yml
Best for Pure-Python projects, web apps, libraries Data science, ML/GPU, cross-language projects
📌 Using pip inside conda

You can mix both: activate a conda env, then use pip install for packages not on conda channels. The environment.yml file has a pip: sub-section for this. Install conda packages before pip packages to avoid solver conflicts.

6 pyproject.toml

pyproject.toml is the modern, unified config file for Python projects (PEP 517/518/621). It replaces the older setup.py and setup.cfg files. It tells build tools everything about your project.

Three sections of pyproject.toml

[build-system]

Which tool to use to build your package (setuptools, flit, hatch, poetry)

[project]

Package metadata: name, version, dependencies, classifiers, URLs

[tool.*]

Per-tool config (pytest, black, mypy, ruff, etc.) — all in one file

pyproject.toml 
# ── BUILD SYSTEM ────────────────────────────────────────────────
[build-system]
requires      = ["setuptools>=68", "wheel"]
build-backend = "setuptools.build_meta"

# ── PROJECT METADATA ─────────────────────────────────────────────
[project]
name            = "my-awesome-package"
version         = "0.1.0"
description     = "A short description of my package"
readme          = "README.md"
license         = {text = "MIT"}
requires-python = ">=3.10"
authors         = [{name = "Alice", email = "alice@example.com"}]
keywords        = ["machine-learning", "data-science"]

# Runtime dependencies (what users need to install your package)
dependencies = [
  "numpy>=1.26",
  "pandas>=2.0",
  "requests>=2.28,<3",
]

# Optional / extra dependencies
[project.optional-dependencies]
dev  = ["pytest>=7", "black", "ruff"]
docs = ["sphinx", "myst-parser"]

# Project URLs
[project.urls]
Homepage   = "https://github.com/alice/my-package"
Issues     = "https://github.com/alice/my-package/issues"

# Command-line scripts
[project.scripts]
my-tool = "mypackage.cli:main"    # → creates 'my-tool' command in PATH

# ── TOOL CONFIGS ─────────────────────────────────────────────────
[tool.pytest.ini_options]
testpaths = ["tests"]

[tool.black]
line-length = 88

[tool.ruff]
line-length = 88
select = ["E", "F", "I"]      # pycodestyle, pyflakes, isort

[tool.mypy]
strict = true

Installing optional extras

# Install the package with dev extras
pip install -e ".[dev]"

# Install multiple extras
pip install -e ".[dev,docs]"
📌 Dynamic version

Instead of hardcoding version = "0.1.0", you can use dynamic = ["version"] and read it from src/mypackage/__init__.py or a version file. Setuptools handles this via [tool.setuptools.dynamic].

7 setuptools

setuptools is the most common build backend — the tool that actually turns your source code into a distributable package (.whl and .tar.gz). It reads pyproject.toml and knows how to find your packages and data files.

How setuptools finds your packages

pyproject.toml — setuptools config 
[tool.setuptools.packages.find]
where   = ["src"]    # look inside the src/ directory
include = ["mypackage*"]  # optional: only include matching
exclude = ["tests*"]      # exclude test directories

# Include non-Python files (data files, templates, etc.)
[tool.setuptools.package-data]
mypackage = ["*.json", "data/*.csv"]

Src layout vs flat layout

Flat layout
my-project/
├── mypackage/
│   ├── __init__.py
│   └── utils.py
├── tests/
├── pyproject.toml
└── README.md

Simple. Risk: accidentally imports the local dir during tests.

Src layout (recommended)
my-project/
├── src/
│   └── mypackage/
│       ├── __init__.py
│       └── utils.py
├── tests/
├── pyproject.toml
└── README.md

Forces proper install. Tests always use the installed version.

MANIFEST.in — what goes into the source distribution

# MANIFEST.in — controls what's in the .tar.gz (sdist)
include README.md LICENSE pyproject.toml
recursive-include src/mypackage/data *.json *.csv
exclude *.pyc
💡 Modern setuptools auto-discovers

With setuptools>=61, you rarely need MANIFEST.in or explicit find_packages(). Just declare [tool.setuptools.packages.find] in pyproject.toml and it auto-discovers everything. Most projects only need a handful of lines.

8 Build & Publish

Building converts source code into distributable artifacts. There are two artifacts: a wheel (.whl) for fast installation, and a source distribution (.tar.gz / sdist) for source transparency.

Source Code
→
python -m build
→
.whl + .tar.gz
→
twine upload
→
PyPI ✓
terminal — build and publish workflow 
# 1. Install build tools
pip install build twine

# 2. Build both wheel (.whl) and sdist (.tar.gz)
python -m build
#   → dist/
#       my_awesome_package-0.1.0-py3-none-any.whl
#       my_awesome_package-0.1.0.tar.gz

# 3. Check the distribution before uploading
twine check dist/*

# 4a. Upload to TestPyPI first (safe testing)
twine upload --repository testpypi dist/*
#   Install from TestPyPI to verify:
pip install --index-url https://test.pypi.org/simple/ my-awesome-package

# 4b. Upload to production PyPI
twine upload dist/*
#   Then anyone can install with:
pip install my-awesome-package

Wheel vs Source Distribution

Wheel (.whl)Source Distribution (.tar.gz)
ContentPre-built, ready to unzipSource code + build instructions
Install speedFast (no build step)Slower (needs to build)
PlatformPure Python: any; C ext: platform-specificAlways platform-agnostic
Filenamename-ver-pyX-none-any.whlname-ver.tar.gz
PurposeWhat users installSource of truth / compliance

Lock Files — Reproducible Environments

A lock file records the exact resolved versions of every dependency (and sub-dependency). requirements.txt generated by pip freeze is a basic lock file. Modern tools like pip-tools or uv handle this better.

pip-tools workflow 
# pip-tools: compile abstract deps → exact lock file
pip install pip-tools

# requirements.in — abstract (what you want)
# numpy>=1.26
# pandas

# Compile to exact versions (reads pyproject.toml or .in file)
pip-compile pyproject.toml -o requirements.txt
# → requirements.txt with exact pinned versions of all transitive deps

# Install from lock file
pip-sync requirements.txt            # syncs env to EXACTLY this

# Update all deps to latest compatible
pip-compile --upgrade pyproject.toml
requirements.in / pyproject.toml

Abstract: "I need numpy>=1.26" — committed to git, human-maintained

requirements.txt (pinned)

Lock: "Use numpy==1.26.4 exactly" — also committed, reproducible builds

9 Full Project Layout

A well-structured Python project that can be installed, tested, and published to PyPI. 

my-project/
├── src/
│   └── mypackage/
│       ├── __init__.py        # makes it a package; can set __version__
│       ├── core.py
│       ├── utils.py
│       └── data/
│           └── defaults.json
├── tests/
│   ├── conftest.py             # pytest fixtures
│   ├── test_core.py
│   └── test_utils.py
├── docs/
│   └── index.rst
├── pyproject.toml            # ← all project config lives here
├── requirements.txt          # lock file (generated by pip freeze / pip-compile)
├── .gitignore
├── LICENSE
└── README.md

Minimal __init__.py

src/mypackage/__init__.py 
from importlib.metadata import version, PackageNotFoundError

try:
    __version__ = version("my-awesome-package")
except PackageNotFoundError:
    __version__ = "0.0.0-dev"   # not installed, running from source

# Public API — only these are exported when someone does `from mypackage import *`
from .core import MainClass
from .utils import helper_function

__all__ = ["MainClass", "helper_function", "__version__"]

Complete pyproject.toml for a real project

pyproject.toml — production-ready 
[build-system]
requires      = ["setuptools>=68", "wheel"]
build-backend = "setuptools.build_meta"

[project]
name            = "mypackage"
dynamic         = ["version"]    # read from __init__.py
description     = "A useful Python package"
readme          = "README.md"
requires-python = ">=3.10"
license         = {text = "MIT"}
authors         = [{name = "Alice", email = "alice@example.com"}]
classifiers     = [
  "Programming Language :: Python :: 3",
  "License :: OSI Approved :: MIT License",
  "Operating System :: OS Independent",
]
dependencies    = [
  "numpy>=1.26",
  "pandas>=2.0",
]

[project.optional-dependencies]
dev = ["pytest>=7", "pytest-cov", "ruff", "black", "mypy"]

[tool.setuptools.dynamic]
version = {attr = "mypackage.__version__"}

[tool.setuptools.packages.find]
where = ["src"]

[tool.pytest.ini_options]
testpaths  = ["tests"]
addopts    = "--cov=mypackage --cov-report=term-missing"

[tool.ruff]
line-length = 88
select      = ["E", "F", "I", "B"]

[tool.mypy]
python_version = "3.10"
strict         = true

Quick-start: New project from scratch

terminal — full new project setup 
# 1. Create project directory
mkdir my-project && cd my-project

# 2. Create virtual environment and activate
python3 -m venv venv && source venv/bin/activate

# 3. Upgrade pip
pip install --upgrade pip

# 4. Create src layout
mkdir -p src/mypackage tests
touch src/mypackage/__init__.py src/mypackage/core.py
touch tests/test_core.py pyproject.toml README.md

# 5. Write pyproject.toml (see section 6/9 above)

# 6. Install your package in editable mode + dev extras
pip install -e ".[dev]"

# 7. Develop, test, iterate
pytest

# 8. When ready to publish
pip install build twine
python -m build
twine upload dist/*
💡 Modern alternatives worth knowing

The ecosystem has faster modern tools: uv (Rust-based, extremely fast pip + venv replacement), hatch (build backend + env manager), poetry (all-in-one with lock file), and pdm. They all still use pyproject.toml as the config format. Learning pip + venv + setuptools first gives you the foundation to understand them all.
