1 Why Test ML Code?

Testing ML systems is fundamentally different from traditional software. Traditional code is deterministic, while ML code often relies on stochastic processes, large datasets, and complex mathematical transformations.

Traditional Software Testing
  • Fixed inputs produce fixed outputs.
  • Unit tests cover discrete logic blocks.
  • Logic bugs are usually explicit crashes.
ML Software Testing
  • Outputs depend on data distribution and model weights.
  • Bugs can be silent (e.g., model trains but metrics are degraded).
  • Need to test data shapes, pipeline steps, and determinism.
💡 The ML Testing Pyramid

In ML, you need Unit Tests (for data transformations and model architecture shapes), Integration Tests (pipeline end-to-end with dummy data), and Validation Tests (checking model metrics against a baseline).

2 pytest Basics

pytest is the standard testing framework in Python. It automatically discovers tests and provides rich output for failures.

Writing a Basic Test

test_preprocessing.py 
import numpy as np
from my_ml_lib.preprocessing import normalize_data

def test_normalize_data_scales_to_zero_one():
    # Arrange
    raw_data = np.array([0.0, 50.0, 100.0])
    
    # Act
    scaled_data = normalize_data(raw_data)
    
    # Assert
    assert np.min(scaled_data) == 0.0
    assert np.max(scaled_data) == 1.0
    assert scaled_data[1] == 0.5

Running Tests

terminal 
# Run all tests in the current directory and subdirectories
pytest

# Run a specific test file
pytest tests/test_preprocessing.py

# Run tests matching a string (e.g., run only 'test_normalize')
pytest -k "normalize"

# Run with verbose output and print statements (useful for debugging ML arrays)
pytest -v -s

3 Fixtures

Fixtures are reusable pieces of setup code. In ML, you often need mock DataFrames, pre-instantiated models, or temporary directories for saving artifacts. Fixtures keep your tests DRY (Don't Repeat Yourself).

Creating and Using a Fixture

test_pipeline.py 
import pytest
import pandas as pd

# Define a fixture
@pytest.fixture
def sample_dataframe():
    """Returns a small DataFrame for testing preprocessing steps."""
    return pd.DataFrame({
        'age': [25, 30, np.nan, 40],
        'income': [50000, 60000, 55000, 120000],
        'target': [0, 1, 0, 1]
    })

# Use the fixture by passing its name as an argument
def test_imputer_removes_nans(sample_dataframe):
    from my_ml_lib import fill_missing_values
    
    cleaned_df = fill_missing_values(sample_dataframe)
    
    assert cleaned_df.isna().sum().sum() == 0
    assert cleaned_df['age'].iloc[2] == 31.66 # Assuming mean imputation
📌 Fixture Scopes

By default, fixtures are recreated for every test function (scope="function"). For expensive ML operations (like loading a heavy pre-trained model for inference tests), use @pytest.fixture(scope="session") to load it only once per test run.

4 Parametrize

@pytest.mark.parametrize allows you to run the same test function with multiple sets of arguments. This is incredibly useful for testing edge cases in data inputs or ensuring a function works across different hyperparameters.

test_activations.py 
import pytest
import numpy as np
from my_ml_lib.activations import relu

@pytest.mark.parametrize("input_val, expected_val", [
    (5.0, 5.0),     # Positive number remains unchanged
    (-2.0, 0.0),    # Negative number becomes zero
    (0.0, 0.0),     # Zero remains zero
    (np.array([-1, 1]), np.array([0, 1])) # Works on arrays
])
def test_relu_activation(input_val, expected_val):
    result = relu(input_val)
    
    # Use np.testing for array comparisons to handle floats and structures
    np.testing.assert_array_equal(result, expected_val)

5 Mocking

ML pipelines often fetch data from databases (SQL, S3) or external APIs. You don't want your unit tests to make real network calls because they will be slow, flaky, and require credentials. Use mocking to simulate these interactions. We typically use the pytest-mock plugin (which wraps unittest.mock).

Mocking an S3 Data Fetch

test_data_loader.py 
import pandas as pd
from my_ml_lib.data_loader import fetch_data_from_s3

def test_fetch_data_formats_correctly(mocker):
    # 1. Create a fake dataframe that the mock will return
    fake_df = pd.DataFrame({'id': [1, 2], 'val': ['A', 'B']})
    
    # 2. Patch the actual network call function
    # Replace 'my_ml_lib.data_loader.boto3_download' with a mock that returns our fake df
    mock_download = mocker.patch(
        'my_ml_lib.data_loader.boto3_download', 
        return_value=fake_df
    )
    
    # 3. Call the function under test
    result = fetch_data_from_s3("s3://my-bucket/data.csv")
    
    # 4. Assert the result and that the mock was called correctly
    assert len(result) == 2
    mock_download.assert_called_once_with("s3://my-bucket/data.csv")
⚠ Mocking where it's used

When patching, you must patch the object where it is imported into and used, not where it is defined. If data_loader.py does from utils import boto3_download, you must patch my_ml_lib.data_loader.boto3_download.

6 Coverage

Test coverage measures the percentage of your code executed during tests. Use pytest-cov to identify untested paths in your ML data processing logic.

terminal 
# Install pytest-cov
pip install pytest-cov

# Run tests and generate a terminal report for the 'my_ml_lib' package
pytest --cov=my_ml_lib

# Generate a detailed HTML report (creates a htmlcov/ index.html)
pytest --cov=my_ml_lib --cov-report=html
💡 Coverage in ML

Aim for high coverage on data preprocessing, feature engineering, and inference API wrappers. Don't worry about getting 100% coverage on complex training loops or third-party library calls. Logic errors in data pipelines are the most common source of ML bugs.

7 Key ML Testing Patterns

1. Determinism (Seeds)

Always set random seeds for numpy, torch, or scikit-learn inside tests or fixtures to ensure tests don't fail randomly due to weight initialization or data splitting.

2. Overfitting a small batch

A classic model test: Create a dataset of 5-10 examples. Train your model on it for many epochs. Assert that the loss goes near zero. If it can't overfit a tiny batch, the architecture or training loop is broken.

3. Shape and Type Checks

ML bugs are often silent broadcasting errors. Explicitly test that your dataset features have the correct .shape and .dtype before and after transformations.

4. np.testing

Never use == for floating-point arrays. Always use np.testing.assert_allclose() or torch.testing.assert_close() to account for minor precision differences.
