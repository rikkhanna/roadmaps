A view is a window into the same memory
When you slice a NumPy array, you don't get a copy of the data. You get a view — a different way of looking at the exact same numbers in memory.
original:
10
20
30
40
50
view [1:4]:
20
30
40
The green cells do not have their own copy of 20, 30, 40. They are looking directly at the same memory slots as the blue cells. Think of it like a window — the window doesn't copy the view outside, it just frames it. 
import numpy as np

original = np.array([10, 20, 30, 40, 50])
view     = original[1:4]          # slice → creates a VIEW, not a copy

print(view)          # [20 30 40]
print(view.base)     # [10 20 30 40 50]  ← points to original
print(view.base is original)  # True
What is .base?
.base is NumPy's way of telling you "who owns the actual memory?" If an array is a view, .base points to the owner. If it owns its own memory, .base is None.
Owns its memory → .base is None
a = np.array([10, 20, 30])
print(a.base)   # None  ← a owns its own memory
Is a view → .base points to the owner
b = a[0:2]        # slice → view
print(b.base)   # [10 20 30]  ← b is looking at a's memory
print(b.base is a)  # True

c = a.reshape(3, 1)  # reshape → also a view
print(c.base is a)    # True
Simple check: array.base is None → it owns its memory.
array.base is not None → it's a view, sharing someone else's memory.
The catch — changing a view changes the original too
Because a view shares the same memory, writing to it writes to the original. Click the button to see it happen.

# This is the surprise that catches everyone:
original = np.array([10, 20, 30, 40, 50])
view     = original[1:3]

view[0] = 999           # you change view...

print(original)  # [10 999 30 40 50] ← original changed too!
print(view)      # [999 30]
⚠️ This is a very common bug in ML pipelines — you slice your training data, modify the slice for processing, and accidentally corrupt the original dataset.
The fix — use .copy() when you want independence
.copy() creates a brand new array with its own memory. Changing it never affects the original.

20

40

view [1:3]

← shared memory (dangerous if modified)
copy [1:3]
20

← own memory (safe to modify)
original = np.array([10, 20, 30, 40, 50])

view = original[1:3]         # view — shared memory
copy = original[1:3].copy()  # copy — own memory

print(view.base is original)  # True  ← view
print(copy.base is original)  # False ← copy owns its data
print(copy.base)              # None

copy[0] = 999
print(original)  # [10 20 30 40 50] ← untouched ✓
✓ Rule of thumb: if you're going to modify a slice, always call .copy() on it first.
What creates a view vs a copy?
Knowing which operations share memory and which don't will save you from silent bugs.
✓ Creates a VIEW (shared memory, no copy)
a[1:4]           # basic slicing
a.reshape(2, 3)   # reshape (usually)
a.ravel()         # flatten (usually)
a.T               # transpose
a[:, 0]           # column selection
✗ Creates a COPY (new memory, safe to modify)
a.copy()          # explicit copy
a[[0, 2, 4]]       # fancy indexing (list of indices)
a[a > 3]          # boolean indexing
np.concatenate([a, b])  # always new array
Quick way to always check:
array.base is None → owns memory (safe)
array.base is not None → it's a view (careful with edits)
np.shares_memory(a, b) → directly check if two arrays share memory 
# The ultimate check
a = np.array([1, 2, 3, 4, 5])
b = a[1:3]
c = a[[1, 2]]     # fancy index → copy

np.shares_memory(a, b)  # True  ← same memory
np.shares_memory(a, c)  # False ← different memory
