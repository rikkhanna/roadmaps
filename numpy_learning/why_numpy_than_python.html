Think of it like two different post offices
Both post offices receive 10 parcels and must stamp each one. But they work very differently.
🐢 Post Office A — Python list
Each parcel arrives wrapped in 5 layers of packaging.
The worker must:
  1. Find the parcel in a big unsorted pile
  2. Read the label to figure out what's inside
  3. Unwrap all 5 layers
  4. Stamp it ← the actual job
  5. Re-wrap it in 5 new layers
  6. Find a new place to store it

Repeat for every single parcel. One at a time.
⚡ Post Office B — NumPy
All parcels arrive bare on a conveyor belt, no wrapping.
A stamping machine just:
  → Stamps each one as it passes

No unwrapping. No re-wrapping. No searching.
The machine runs at full speed the whole time.
The actual work — the stamp — takes the same time in both offices. The difference is all the extra steps Post Office A does around it. In Python, those extra steps are called overhead.
Watch it happen — same 8 numbers, two approaches
Both need to multiply 8 numbers by 2. Watch how differently they work.
What exactly is the "wrapping" in Python?
Every number in a Python list is not just a number — it's a full Python object. Here's what that object actually contains in memory.
The number 42 stored in a Python list:
📦 PyObject — the wrapper around every Python number
ob_refcnt
Reference count — how many things point to this object (for garbage collection)
ob_type
Pointer to the type — e.g. "I am an int, not a float or string"
ob_digit
42 ← the actual number. Finally.
The number 42 stored in a NumPy array:
⚡ 8 bytes of raw memory 42 ← just the number, nothing else
When Python does x * 2, it first has to read ob_type to check "can I multiply this?", then dig down to ob_digit to get the actual number. For 1 million elements, that's 1 million type checks Python didn't need to do — NumPy already knows every element is a float64.
There's another reason — where things live in memory
Your CPU has a small fast storage called cache. If the next number is right next to the current one in memory, the CPU already has it loaded. If it's far away, the CPU has to wait.
A Python list stores pointers (addresses) that point to objects scattered all over memory. Every element access is a "go fetch" across a long distance.

A NumPy array stores the actual numbers packed tightly, side by side. The CPU loads a whole chunk at once. By the time it finishes one number, the next 7 are already sitting in cache, ready. This is called cache locality.
So why is NumPy faster? One sentence:
Python spends most of its time on wrapping, unwrapping, and looking up each number — NumPy skips all of that and just does the math.
Python list: numbers are wrapped in objects  →  unwrap, type-check, do math, re-wrap  →  × 1,000,000 times
NumPy: bare numbers in one block of memory  →  C loop does the math  →  done
The math itself (1 × 1 multiply) takes the same time in both. The difference is everything around the math.
